# OceanFishy

A single-page ocean/marine conditions and fishing-forecast web app for Australian waters. Live at https://ozrocozroc.github.io/OceanFishy/

## Architecture

- **Entire app is one file: `index.html`** (~3000 lines) — HTML, CSS, and JS all inline, no build step, no dependencies beyond Leaflet (CDN) and the fetch APIs below. This is intentional; don't split it up without being asked.
- Hosted on **GitHub Pages**, serving directly from `main`. `index.html` is the entry point (renamed from `ocean-monitor.html` so it serves at the bare root URL).
- No backend. Every data source below is called directly from the browser.

## Data sources

| Feature | Source | Notes |
|---|---|---|
| SST / Chlorophyll / Altimetry / Depth map layers | NOAA/NASA ERDDAP | Real satellite data, has a real lag (SST ~1-2 days, Chl VIIRS ~2-3 weeks, Chl OLCI ~2-3 days, Altimetry ~3-5 days). The "Latest" date chip queries ERDDAP's own `(last)` time index — it's not guessed. |
| Currents overlay | NOAA ERDDAP (`noaacwBLENDEDNRTcurrentsDaily`) | No WMS support, rendered client-side from raw u/v vectors via a canvas particle animation. |
| Forecast panel (temp/rain/wind/pressure) | Open-Meteo weather API | Model is user-selectable, see below. |
| Forecast panel (wave/swell/tide) + Wind/Swell map overlays | Open-Meteo marine API | Wave model is user-selectable, see below. Tide is **always** Best match — no other model provides `sea_level_height_msl` at all. |

**No API keys anywhere.** Everything above is either a free public API or a direct unauthenticated fetch.

## Forecast model selection (important — read before touching `openForecastPanel`)

The forecast panel has two independent model pickers, each with a capability-flagged config array (`MARINE_WAVE_MODELS`, `WEATHER_MODELS`) and a per-field auto-fallback pattern:

- **Wave model** (`marineWaveModel`, default `ecmwf_wam`) — confirmed by direct comparison to track Windy's own forecast closely; Open-Meteo's auto "best_match" ran wave height/power ~15-30% low at the one location tested.
- **Weather model** (`weatherModel`, default `best_match`).
- Both persist via `localStorage` and trigger a live re-fetch of the open panel on change.

**The fallback policy, and why it exists:** not every model provides every field, and the gaps are *not* what you'd guess from a null-check alone:
- Tide (`sea_level_height_msl`) — **only** `best_match` provides it. Every named wave model returns `null`, confirmed directly against the API.
- Swell decomposition — `ecmwf_wam`/`ecmwf_wam025` return `null` for it (expected). But `ncep_gfswave025` and `gwam` return **non-null** swell values that are numerically **identical** to `wave_height` on every sample checked — i.e. they alias swell to total wave rather than truly decomposing it. Only `best_match` and `meteofrance_wave` were confirmed to return genuinely distinct swell data. **Don't just check for `null` when adding/auditing a model — compare the actual values.**
- Pressure (`pressure_msl`) — `ncep_gfs013` (NOAA GFS 0.13°) returns `null` for pressure specifically, even though temp/wind/rain work fine on the same model.
- `bom_access_global` (Australia's own BOM model) was tested and excluded entirely: its `data_end_time` (check via `https://api.open-meteo.com/data/{model}/static/meta.json`) is over a year stale. Worth re-testing if Open-Meteo's BOM feed comes back online.

When a selected model is missing a field, the app fires a small **extra fetch to `best_match` for just that field** and merges it in — the row never silently prints all `-`. `-` per cell only appears if even `best_match` has no data for that specific hour (a genuine data gap, not a model-choice artifact). See `openForecastPanel` for the merge logic — it's the reference pattern for adding any future per-field fallback.

**Before adding a new model to either list:** verify it against the live API directly (`curl`/`python -c` a real request), check the field values aren't aliased duplicates of another field, and check its `meta.json` isn't stale. Marine model parameter names are inconsistent and undocumented in places (e.g. `ecmwf_wam` not `ecmwf_wam9`, `gwam` not `dwd_gwam`) — don't guess, test.

## Timezone conventions

- **Forecast panel** (day headers, hour labels, cursor tag, tide High/Low times) — shown in the **viewer's local timezone**, via plain `Date` getters (`getHours()`, `getDay()`, etc.), not `getUTC*()`. Day-grouping for the header row also buckets by local calendar day (`localDayKey` helper), not UTC date — an hour that's technically "tomorrow" in UTC but still today locally must show under today's header.
- **Date chips / satellite layer dates** — the chip **label** (weekday, day/month) is local time, but the **value actually requested from ERDDAP** stays UTC always (`iso`, `curDate`, the `setUTCDate` stepping). Never let these drift — a chip must always request the date its label implies.
- **Live Readings coordinate line** — same treatment as the date chips: the label converts `curDate` to the viewer's local calendar day (local `Date` getters on a UTC-midnight anchor), with the real UTC date kept in a `title` tooltip (`Dataset date (UTC): …`) for anyone cross-checking against the active chip. Changed from an earlier version that printed the bare UTC date suffixed `(UTC)` directly in the line — per user request, to match how the chips/forecast panel already read.
- Anywhere else a bare UTC date/time is still shown without conversion (e.g. the currents lag status message), it's explicitly suffixed `(UTC)` so it's never mistaken for "today."
- `todayStr()` is only a placeholder shown briefly before `useLatestAvailableDate()` resolves (or if that fetch fails offline) — it subtracts via absolute milliseconds (`Date.now() - 2*86400000`), not local `setDate()`, so it's timezone-independent.

## Fronts layer (frontal-strain proxy, not real FSLE)

- Real FSLE (Finite-Size Lyapunov Exponent, like TimeZero's product) requires an AVISO/CNES API key — confirmed no free/keyless source exists (checked NOAA CoastWatch/PolarWatch/AOML ERDDAP, `tds.aviso.altimetry.fr`, Copernicus — all require login). Instead the "Fronts" layer (`b-fsle`, layer key `fsle`) computes a **strain-rate proxy** from real current u/v data (`noaacwBLENDEDNRTcurrentsDaily`): central-difference normal + shear strain, `strain = sqrt(normal²+shear²) * 86400` (1/day). It is explicitly labeled "(proxy)" everywhere in the UI — never claim it's real FSLE.
- Calibration (EAC region, checked directly against the API): median ~0.66/day, p90 ~1.5/day, p95 ~1.9/day, p99 ~2.8/day. `FSLE_STRAIN_STOPS` color ramp is tuned to these; values below the first stop (0.3/day) return `null` (fully transparent) on purpose so the layer reads as ridge lines, not a full wash. Red/high strain = strong convergence (where drifting bait/debris concentrate), not an unconditional "better fishing" score — same caution-flag logic as the extreme-weather marker, not a bonus signal.
- `createCanvasRasterLayer(layerKey, colorFn, maxPts, fetchGrid, hasAlpha)` gained a `hasAlpha` param so `colorFn` can return `[r,g,b,a]` instead of `[r,g,b]`; Bathymetry/SST stay on the original flat-opaque path, only `fsle` uses it.
- **Smooth rendering technique** (the one that actually worked, after two failed iterations — flat rgba quads had double-alpha-stacking seams, then an opaque-mosaic + `destination-in` mask fixed the seams but flat quads still had hard color *edges*): each grid quad is drawn as a **2x2-pixel micro-texture** (`makeMicroTexture()`) holding the four corner color values, upscaled onto the quad's screen rect via `ctx.drawImage()` with `imageSmoothingEnabled=true`. The browser's bilinear upscale is mathematically correct interpolation across the whole quad, and adjacent quads share exact corner values so colors blend continuously with no visible grid. This is now the reference pattern for any future translucent gridded raster layer.
- **Known open issue (unverified on real device):** a real-phone screenshot showed the Fronts layer rendering as a rigid grid of disconnected squares with gaps — not reproducible via `resize_window` viewport emulation in this environment (that only changes CSS viewport, not real device GPU/canvas/WebKit behavior, so don't trust emulated-viewport testing as conclusive for canvas rendering bugs). Two fixes attempted: (1) removed SUBDIV=4 bilinear supersampling of the u/v grid — didn't fix it per user report; (2) replaced the single reused `microCanvas`/`microCtx`/`microImg` (mutated via `putImageData` before each `drawImage` call in a loop) with a **fresh canvas created per quad** (current code), on the theory that some mobile WebKit versions can serve a stale snapshot of a canvas repeatedly reused as a `drawImage` source. Pushed but not yet confirmed fixed on a real device — if reported broken again, get a fresh real-device screenshot rather than trying to repro via emulation.

## Hotspot finder scoring

`updateHotspots(bounds)` combines SST temp-break gradient, Chl-a front gradient, bathymetry-edge gradient, SLA front gradient, current speed, frontal strain, and a warm-core eddy bonus: `WEIGHTS = {sst: 0.22, chl: 0.18, depth: 0.18, sla: 0.12, strain: 0.12, current: 0.10, eddyCore: 0.08}`. Tuned specifically for pelagic gamefish (marlin/tuna/mahi) targeting, not generic "any fish" structure-finding — per explicit user request.

- **Current speed and frontal strain are not redundant despite sharing the same source fetch.** `fetchHotspotCurrentGrid` returns both `grid` (speed, kt) and `strainGrid` (via the shared `computeStrainGridPerDay`, reused from the Fronts layer to avoid a second network round trip) from one request. But they measure different things: current speed (`hotspotCurrentScore`, bell curve peaking at 0.7kt) is a *workability gate* — is there enough flow to hold a berley trail without being unfishable — while strain is a *spatial gradient* of the velocity field — is the water actually converging/shearing right here. A broad uniform fast current can have high speed but low strain (good drift, nothing concentrating); a sharp eddy edge can have only moderate speed but high strain (classic bait-holding front). Same reasoning as why SST itself isn't scored, only its gradient is.
- **Warm-core eddy bonus (`eddyCoreScore`)** — separate from the SLA *gradient* term (front steepness). Gamefish concentrate on/around the eddy core itself, not just its edge, so a candidate near a strong positive SLA extremum scores higher even with a modest local gradient. Only positive (warm) SLA is rewarded — cold cores/upwelling matter for bait/plankton but aren't the pelagic-gamefish signal this targets. Scaled off the same `EDDY_CORE_THRESHOLD` (30cm) used by the warm/cold-core map badges (`eddyCoreIcon`), so "full bonus" lines up with what's already flagged worth-fishing on the map.
- **SST-range bonus** — folded into the `sst` weight itself (`0.7 * gradientNorm + 0.3 * sstRangeScore(...)`), not a separate weighted slot, to avoid growing the popup breakdown row. Rewards water temp sitting in a productive band for whichever gamefish are in season (`activeGameSeasons(month)`, shared with `suggestSpecies` so the two windows never drift apart) — a sharp temp break at 14°C scores well on gradient alone but is useless for marlin/mahi, which this corrects for.
- **Chl-a clear-side bonus** — also folded into the `chl` weight (`0.7 * gradientNorm + 0.3 * clearSideNorm`). Real gamefishing wisdom is "fish the color change but stay on the clean side" — a candidate deep inside high-chl (turbid) water can have the same local gradient as one right on the clear-water edge, so `chlClearSide` (own chl value vs. neighbor average) biases toward the low-chl side of the front.
- Extreme-weather warning marker thresholds: `EXTREME_WIND_KMH = 35, EXTREME_SWELL_M = 2` (tuned down twice from an initial 54km/h/3m based on user feedback that it was too conservative).
- **Score is honestly capped when a source is down, not rescaled to 100.** `rawScore` renormalizes against each *candidate's own* weightSum (correct for per-cell noise, e.g. one candidate happens to sit under Chl cloud cover — shouldn't penalize it relative to a clear-sky candidate nearby). `sourceWeightCap` is a separate, run-wide constant (sum of weights for sources that actually responded this call) that then scales the *displayed* score down — so an SST+depth-only run (40% of total weight) shows at most ~40/100, not rescaled up to look like a full 7-signal run. Don't collapse these back into one renormalization — that reintroduces the exact bug this fixed (confirmed live: an SST+depth-only run was showing 82/100 before this, now correctly shows ~33/100 "capped at 40").
- **Per-source fault tolerance:** SST and bathymetry are hard requirements — they drive the search grid and the land/depth filter respectively, so their fetches are NOT caught, and a failure there aborts the whole run via the outer `catch` (existing behavior). SLA, current (+strain), and Chl-a are each wrapped in `.catch(() => null)` at the `Promise.all` call — if one of those sources fails entirely (confirmed live: a real corsproxy/ERDDAP hiccup during testing took out all three at once), its signal just drops out of scoring for every candidate and the remaining weight redistributes, same mechanism Chl's per-cell cloud-gap handling already used. The status bar names which sources were unavailable (`"...SLA/currents/Chl-a unavailable — scoring adjusted"`). Don't reintroduce a single unguarded `Promise.all` for all 5 fetches — that's the exact bug this fixed (one source outage silently killing the whole tool).

## NOAA ERDDAP outages

`coastwatch.noaa.gov` has gone down before with `FileSystemException: No space left on device` on the server side (confirmed via direct curl on the WMS endpoints) — a real third-party outage, not an app bug. When a layer looks "broken," curl the WMS endpoint directly and read the XML `ServiceExceptionReport` before assuming it's a code issue. Separately, `loadOceanLayer()`'s Leaflet `'load'` handler was unconditionally clearing `setStatus('')` once tiles settled (success or fail), silently erasing the `tileerror` handler's error message almost immediately — fixed with a `hadTileError` flag so a genuine outage still shows its error text instead of looking like a silently blank layer.

## UI/layout gotchas already fixed here (don't reintroduce)

- **Any floating panel positioned inside `#map`** (the Leaflet container) will have its clicks bubble up to `map.on('click')` unless you call `L.DomEvent.disableClickPropagation(el)` on it. This bit us three times (forecast panel, sidebar, mobile drawer toggle) before the pattern was applied everywhere. If you add a new floating control inside `#map`, apply this immediately.
- **z-index ties fall back to DOM order.** `#sb` and `#src` (the bottom attribution bar) were both `z-index:900`; `#src` came later in the markup and silently won, covering the sidebar's bottom edge. When two absolutely-positioned siblings might overlap, give the one that should always win a clearly higher z-index, don't rely on source order.
- **CSS specificity: `#parent .child { }` (ID + class) beats a bare `#child { }` (ID only).** Cost real time twice this session (marks-panel flex-shrink, mobile drawer). If a plain-ID override isn't taking effect, check whether a `#ancestor .class` rule elsewhere outranks it, and either raise specificity (`#ancestor #child`) or restructure.
- **An inline `style="..."` attribute always beats an external/`<style>`-block rule, regardless of selector specificity.** Most elements here are built with inline styles (template-literal markup), so a mobile `@media` override that silently does nothing is usually this, not a specificity or load-order issue — check the element's own `style=` attribute for the property you're trying to override before assuming the selector is wrong, and use `!important` if it does declare that property inline (e.g. `display:none!important` was needed to hide the forecast-panel header bar on mobile, since it carries an inline `style="display:flex;..."`).
- **Flex children need an explicit `min-height`/`flex-shrink` override to actually shrink** below their content's natural size — the sidebar's Known Marks panel silently refused to shrink until this was set explicitly on it and its scrollable child.
- **`vh` units don't track a mobile browser's actual visible viewport** (address bar collapse/expand) — use `dvh` (with a `vh` fallback line before it for older browsers) for anything anchored to the bottom of a mobile screen.
- The `preview_screenshot` tool is flaky in this environment (frequently times out) — prefer `preview_inspect` / `preview_eval` + `getComputedStyle` for verifying layout/color, they're reliable here. If a real screenshot is needed and the tab is freshly-navigated, `document.hidden`/`visibilityState` can report `'hidden'` even on an "active" tab (causing `map.getSize()` to return `{0,0}`) — fall back to an already-rendered tab, composite the canvas via `canvas.toDataURL()`, and extract the base64 string via a saved tool-result file if it's too long to return directly.
- Forecast panel day columns are tinted brighter (`nightBg`/`nightBgSticky`, `rgba(255,248,225,.75)` translucent / `#f3ead0` opaque sticky) with `hourLabelColor(isDay)` switching hour-label text to dark for contrast — night columns stay dark. Keep both in sync if adjusting one.

## Deployment

- Push to `main` → GitHub Pages auto-deploys.
- **The deploy step fails intermittently** (build succeeds, deploy step fails) — this is GitHub-side flakiness unrelated to code changes, confirmed by seeing the exact same commit succeed on a bare retry. If a deployment run fails or never appears at all after a push, push an empty commit (`git commit --allow-empty -m "Retry Pages deployment"`) and check again. Usually resolves within 1-2 retries.
- To verify a deployment: `curl` the live URL with a cache-busting query param (`?nocache=$(date +%s)`) and grep for a string unique to the just-pushed commit.
